# Fast Review Mode — Completed Architecture

Fast review is now the production reasoning path for ReviewForge.

## Architecture

The physical pipeline remains deterministic:

```text
FetchPrMetadata → PrepareRepository → ExecuteReasoningEngine → PostToAdo
```

`ExecuteReasoningEngineStage` selects the engine from `Config.reasoning_engine`.

- `single_pi` is the default. It performs intent reconstruction, context gathering, review, verification, and severity calibration in one Pi invocation.
- `multi_stage` is an explicit legacy fallback for debugging, benchmarking, regression comparison, and emergency recovery.

Python owns orchestration, validation, artifact generation, and Azure DevOps side effects. Pi reads repository context and returns JSON only.

## Canonical contract

Pi returns `ReviewResult` directly. `FastReviewResult` and `FastReviewStage` are not production concepts.

`ReviewResult` contains:

- review and verification summaries
- PR intent and risk summary
- findings with observation, impact, recommendation, severity, confidence, and evidence
- symbols, changed lines, related files, tests, and work-item evidence
- discarded findings, positive observations, uncertainties, and metrics

Run metadata (model, engine, duration, and token usage) is added by Python because Pi cannot reliably know those values.

## Compatibility artifacts

The single-call engine synthesizes the established intermediate artifact files from `ReviewResult` so existing diagnostics and post-only workflows continue to work. `review-result.json` is the canonical domain output. `final-findings.json` is generated by `reviewforge.pipeline.projection` for the existing ADO posting contract.

The raw Pi response is written under the run's `raw/` directory for diagnostics and is not part of `ARTIFACT_NAMES`.

## Configuration

```text
REASONING_ENGINE=single_pi   # default production path
REASONING_ENGINE=multi_stage # explicit legacy fallback
FAST_REVIEW=1                # backwards-compatible single_pi alias
```

There is no automatic fallback from `single_pi` to `multi_stage`; failures are surfaced by the orchestrator so operators can choose the fallback explicitly.
