1. What running run-open-prs.ps1 actually does

The script is the batch entrypoint. It does not post comments itself. It enumerates PRs and spawns one short-lived container per PR; that container talks to Azure DevOps, runs the model, and posts findings. From the user's seat the flow is:

1
Discover
Lists active (non-draft) PRs across the configured projects whose target branch matches ADO_TARGET_BRANCHES. Parallel via ForEach-Object -Parallel (PS 7+), sequential fallback for PS 5.1. az repos pr list --status active paginates 100 at a time.
2
Select
Auto-prompts in a TTY (1,3-5, all, none). Skips when stdin/stdout are redirected (CI, scheduled runs).
3
Build (optional)
If -Build is passed (or you call build.ps1 directly), runs docker build / podman build with PI_VERSION pinned via build-arg.
4
Spawn
For each selected PR, calls run.ps1 -Org -Project -RepoId -PrId -AdoToken. That script picks a container runtime (docker wins, podman fallback), mounts ~/.pi/agent/auth.json if present, and starts a detached container (-d --rm).
5
Report
Tracks per-PR exit codes. Final output: "Completed N pull request review run(s)." or "ERROR: N pull request review run(s) failed." with non-zero exit.

One PR = one container. The script will not start a second review for a PR whose container is still running. Containers are detached, so the script's exit code reflects the spawn result, not the review itself.

2. Per-PR lifecycle, end to end

Inside the container (/app/scripts/main.py → auto_pr_reviewer.cli.main), the pipeline runs 12 fixed stages in order. Each stage writes a JSON artifact; failures short-circuit the run.

#StageDoesWrites
1FetchPrMetadataStagePulls PR, work items, comments, threads from ADO REST.metadata.json, work-items.json, threads.json
2PrepareRepositoryStageResolves source/target branches via ADO, clones, produces unified diff.diff.patch, changed-files.json, commits.txt
3BuildArtifactsStageMaterialises the system prompt with the LANGUAGE directive appended.review-system.combined.md
4ReconstructIntentStagePi call: infer author intent (work items, commit history, diff).intent.json
5PlanContextStagePi call: what extra context (files, searches) is needed?context-plan.json
6CollectContextStageResolves file slices and ripgrep matches from the local clone.collected-context.json
7ContextDigestStagePi call: condense the collected context into a usable digest.context-digest.json
8ReviewDiffStagePi call: produce candidate findings. Diff chunked at MAX_DIFF_BYTES; per-chunk outputs under raw/.candidate-findings.json
9VerifyFindingsStagePi call: cross-check each candidate against the digest to drop false positives.verified-findings.json
10CalibrateSeverityStagePi call: assign final severity & produce review doc.severity-findings.json, final-findings.json
11AcceptanceCriteriaCoverageStageDeterministic heuristic: for each linked work item's acceptance criterion, checks whether its identifiers (file paths, function names) appear in the diff. Uncovered ACs become general-thread findings.appends to final-findings.json
12PostToAdoStageDedupe, post new findings as ADO threads, update vote. Also annotates prior bot threads whose (file, line) is no longer in the current diff with a "stale" comment.posted-comments.json

Stages 4–10 each invoke the pi CLI as a subprocess. ADO credentials are stripped from the Pi subprocess env by ai/runner.py so the model cannot call ADO directly. All ADO traffic lives in auto_pr_reviewer.ado.* and is reachable only from stage 12.

3. What you'll see in the PR

When the run posts (i.e. not DRY_RUN=1):

  • One comment per finding. File-anchored findings become inline PR comments on the line ReviewDiffStage chose; unanchored findings (e.g. work-item scope gaps) become general PR comments.
  • Each comment ends with a marker line of the form prb:<12-char-key>. The key is sha1(file|line|severity|title|message)[:12]. This marker is how the bot recognises its own comments on the next run.
  • Vote. If VOTE_WAITING_ON is set to a severity (e.g. major) and any finding meets it, the bot casts waiting for author. Other reviewers' threads are never touched.
  • No review summary comment when there are zero findings — only the file-anchored threads, if any.
  • Acceptance-criterion gaps (linked work items only): if a work item's AC text mentions file paths, class names, or function names that are nowhere in the diff, the bot posts a general-thread finding titled Work item #N acceptance criterion not covered: .... False positives are possible (heuristic, no LLM) — read the message, then dismiss if it's noise.
  • Stale annotations on prior bot threads: when the source branch moves on, the bot appends a follow-up "🤖 stale — this finding no longer anchors..." comment to any of its previous inline findings whose (file, line) is no longer in the current diff. Threads are not deleted or resolved.
Work-item findings always post as general comments. Even if the model "helpfully" guessed a file, the posting path strips file/line so the finding reads as PR-level scope commentary, not as an inline blame. The title prefix is Work item #N .... AC-coverage findings use the same prefix and follow the same general-thread rule.

4. Where the run output lives

Every run writes to artifacts/pr-<PR_ID>/runs/<RUN_ID>/ (volume-mounted at /workspace/artifacts). latest.txt in pr-<PR_ID>/ points at the most recent run.

FileUse it for
final-findings.jsonAuthoritative review payload — same shape posted to ADO.
posted-comments.jsonWhat the bot actually posted (thread id, marker, URL).
run-summary.jsonPer-stage status, duration, token usage. Redacts secrets.
raw/Per-stage Pi outputs, including per-chunk review turns.
diff.patchUnified diff the model reviewed (what it actually saw).
intent.json, context-plan.json, context-digest.jsonIntermediate reasoning — useful when a finding looks off and you want to retrace the model's path.

The artifact tree is the ground truth for "what did the bot do". When ADO shows a comment you disagree with, open raw/review_diff_chunk_*.json and final-findings.json to see why.

5. Idempotency: re-running the bot

Re-running run-open-prs.ps1 on the same PR is safe.

  • Each comment carries the prb:<key> marker on its own line. Stage 11 scans existing threads, computes the marker for each candidate finding, and skips any match.
  • Stable dedupe fields: file, line, severity, title, message. Fields excluded: suggestion, contextBasis, confidence, severity_calibration, timestamps. Reruns that don't move those five fields will not double-post.
  • Pi session reuse is on by default: the session id pr-<id>-review-<run_id> retains the system prompt, diff, and prior turns between stage calls, which keeps the per-PR token cost predictable. Set PI_SESSION_ENABLED=0 for deterministic reruns.
  • To force a fresh session under the same id (e.g. after a schema change), pass --pi-session-clear.
Marker regex is anchored: ^prb:([a-zA-Z0-9]{6,32})$. If you ever paste a review into the same PR by hand, do not put prb: at the start of a line — you would confuse dedupe.

6. Exit codes and "did it work"

ExitSourceMeaning
0Python orchestratorAll stages OK or SKIPPED. Post either succeeded or was skipped (DRY_RUN).
1Python orchestratorAt least one stage returned FAILED. Read run-summary.json for which one.
2CLIConfig error — missing required env var (token, org, project, repo, pr_id). validate-config surfaces the exact key.
127Container runtimeContainer didn't start (image missing, CRLF on scripts/main.py, port, etc.). Check the spawn output.

The PowerShell wrapper returns the container's spawn exit code; per-PR results are independent. run-open-prs.ps1 counts failures and exits non-zero if any PR failed.

7. Triage playbook

The bot posted a finding I disagree with

  1. Open final-findings.json for that PR. Read the finding's message and contextBasis.
  2. Cross-check against intent.json and context-digest.json to see what the model had in scope.
  3. If the finding is correct but the title/severity is off, edit the JSON and re-run python -m auto_pr_reviewer post --input final-findings.json --pr <PR> to re-post. Same dedupe key = same marker = no duplication.
  4. If the finding is wrong, reply on the PR with reasoning. The bot does not currently delete or resolve its comments on rerun — see Known limits below.

Re-run produced duplicates

The marker regex must match on its own line, last line of the comment body. Open one of the bot's comments and verify the last line is exactly prb:<12 chars>. If your custom template (see COMMENT_TEMPLATE_PATH) appends text after the marker, fix the template — markers must be the very last line.

Container never starts

  1. docker images | grep pr-review-bot — did build.ps1 actually run?
  2. Look for python3\r in the error — a Windows CRLF on scripts/main.py shebang. The Dockerfile strips CRLF; if you build outside Docker, run find ./scripts ./src -type f -name '*.py' -exec sed -i 's/\r$//' {} +.
  3. Pull az extension list and verify azure-devops is installed.

Stage 11 didn't post

Either DRY_RUN=1 or every finding collided with an existing marker. Check posted-comments.json — empty array means dedupe said "already up".

Pi timed out

Default PI_TIMEOUT_SECS=600. The stage that timed out is recorded in run-summary.json. Increase the cap, or split the diff with MAX_DIFF_BYTES. Diff > 200 KB auto-switches to file-based chunking unless DISABLE_CHUNK_REVIEW=1.

Known limits (out of scope)

  • Bot threads are not deleted or resolved when the PR changes. Re-runs append a "stale" annotation (see §3) but never delete or resolve prior threads — the append-only comment trail is intentional for audit purposes.
  • Open-prs auto-discovery is PowerShell-only. The Python open-prs subcommand intentionally fails fast with a pointer to ./run-open-prs.ps1.
  • Single-file diffs larger than the chunk budget are partially truncated; consider DISABLE_CHUNK_REVIEW=0 + lower MAX_DIFF_BYTES.
  • AC coverage is a deterministic identifier-matching heuristic, not an LLM assessment. False positives are possible — dismiss in the PR if a finding doesn't apply.
