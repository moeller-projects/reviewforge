1. What running the bot actually does explanation

ReviewForge is a Dockerized reviewer for Azure DevOps pull requests. It reads the PR diff, runs the Pi coding agent against it (read-only), and posts any findings as ADO comment threads. It does not merge, block, or change code. By default it does not vote; it only comments.

Single PR

./run.ps1 -PrUrl "https://dev.azure.com/contoso/Payments/_git/payments-api/pullrequest/1423"
Spawns one container for that PR. Best for ad-hoc review or debugging.

Batch (all active PRs)

./run-open-prs.ps1
Discovers active PRs, then loops run.ps1 for each one. Each PR gets its own container.

Dry run (no posting)

./run.ps1 -PrUrl "https://..." -DryRun
Produces final-findings.json but does not write to ADO. Useful for tuning prompts or checking config.

Scheduled/unattended

./run-open-prs-scheduled.ps1 -MaxPullRequests 5
Logs to a transcript and runs the batch script. The separate setup-open-prs-schedule.ps1 registers a Windows Task Scheduler job.
Important
The PowerShell scripts do not load .env automatically. You must populate the live process environment (e.g. set -a; source .env; set +a in bash or direnv in PowerShell). The script forwards those values into the container.

3. Per-PR lifecycle, end to end reference

For a single PR, the container runs these 12 stages in order:

StageWhat it does in plain languageYou care because...
1. Fetch PR metadataReads title, description, reviewers, linked work items, and existing comment threads from ADO....the bot knows whether the PR is linked to work items and can detect already-posted comments.
2. Prepare repositoryClones the source branch and computes the diff against the target branch....the review is based on the actual diff, not just the PR description.
3. Build artifactsAssembles the reviewer's instructions, standards, and language directive into one system prompt file....this is the "personality" that will be loaded into Pi.
4. Reconstruct intentAsks the model: "What is this PR trying to do?"...later stages judge the code against the author's intent, not against the diff alone.
5. Plan contextDecides which other files to read and which tests to look for....the review can reference broader context (e.g., schema definitions, service interfaces) without being fed the entire repo.
6. Collect contextReads the planned files and runs the planned searches (e.g., ripgrep) inside the container....the model gets only the context it asked for, staying under token limits.
7. Context digestSummarises the collected context into a compact digest....the review stage has a focused, readable background instead of raw file dumps.
8. Review diffThe main review. The model inspects the diff and outputs a JSON list of findings. Large diffs are split into chunks....this is where most of the comments come from.
9. Verify findingsFor each finding, asks the model to argue against itself and drop false positives....fewer noisy comments in your PR.
10. Calibrate severityReclassifies each finding's severity (nit / minor / major / blocker)....the comment thread shows a stable, comparable severity label.
11. AC coverageScans linked work-item acceptance criteria to see if any are missing from the diff. Optionally asks the LLM for a second opinion....you may see a general PR comment listing uncovered acceptance criteria.
12. Post to ADOPosts or skips comments based on the dedupe marker, optionally casts a "waiting author" vote, and annotates stale comments from earlier pushes....this is the only visible outcome. It is skipped entirely in dry-run.
Pipeline mode shortcuts
The default ./run.ps1 path runs all 12 stages. Two subcommands exist for special cases: reviewforge review --review-only (stages 1-11, no ADO write) and reviewforge post --input final-findings.json --pr 12345 (only stage 12). You rarely need these directly; the PowerShell wrappers handle them.

4. What you'll see in the PR explanation

Inline comments

Most findings appear as inline comment threads on the changed file. Each thread body contains:

  • A severity label (e.g., blocker, major, minor, nit).
  • A short title and a message explaining the issue.
  • Optional: a suggestion block, confidence label, and context basis (diff-only / surrounding-code / full-module-review).
  • A last line like prb:abc123def456. This is the idempotency marker. Do not edit or delete it.

General PR comments

Two kinds of findings become top-level PR comments instead of inline threads:

  • Work-item coverage findings (e.g., "Work item #1234 — acceptance criterion 'X' not reflected in the diff") are deliberately posted as general comments.
  • Any finding where the model could not map the line number to a valid diff anchor falls back to a file-level comment.

Stale annotations

If you push a new commit, the bot may append a reply to its old comment threads: "🤖 stale — this finding no longer anchors to a diff line in the latest commit." It does not resolve or delete them automatically.

Votes

By default, the bot does not cast a vote. If the operator has configured VOTE_WAITING_ON=major (or another severity), the bot will cast a Waiting for the author vote when the PR has a finding at or above that severity. The vote is updated on every run, not just the first one.

5. Where the run output lives reference

Every run writes a directory tree on the host. The root is either the path set by ARTIFACT_PATH or the named Docker volume reviewforge-artifacts.

artifacts/pr-<PR_ID>/runs/<RUN_ID>/
  metadata.json
  diff.patch
  changed-files.json
  commits.txt
  intent.json
  context-plan.json
  collected-context.json
  context-digest.json
  candidate-findings.json
  verified-findings.json
  severity-findings.json
  final-findings.json
  posted-comments.json
  run-summary.json
  review-system.combined.md
  raw/

The single best file to inspect when something is off is run-summary.json. It contains each stage's status, duration, token usage, and any error. If a stage failed, the details include the exception.

6. Re-running the bot: idempotency explanation

Re-running the same PR does not create duplicate comment threads. Before posting, the bot checks whether a marker like prb:abc123def456 already exists in any existing bot thread on the PR. If the marker is found, the finding is skipped.

Do not edit the marker line
If you edit the last line of a bot comment, the next run may not recognize it as already posted and will create a duplicate. It is safe to reply to the thread, resolve it, or mark it Active.

The marker is computed from file|line|severity|title|message. If you change any of those fields (e.g., by rewording a custom prompt or by the model producing a different message), the bot will see a new key and post again. This is intentional: it allows the reviewer to evolve.

7. Exit codes and "did it work?" reference

./run.ps1 reports the exit code of the container it spawned. The container exit code is produced by python -m reviewforge:

CodeMeaningWhen to expect it
0Review completed.All stages finished (some may be skipped). Comments may have been posted or skipped. Dry-run also returns 0.
1Unexpected failure.A stage failed (Pi error, JSON parse, ADO network failure, git clone failure). Check run-summary.json.
2Configured severity threshold hit. FAIL_ON=minor (or major/blocker) was reached by at least one finding. The comments were still posted.
3Configuration error. Missing required env or invalid values.

For a batch run, the wrapper loops over PRs and continues even if one container fails. The transcript for run-open-prs-scheduled.ps1 captures per-PR container IDs and exit codes.

8. Triage playbook how-to

SymptomWhat to checkLikely fix
Bot comments keep duplicatingOpen the comment and check that the last line is prb:<12 chars> and is on its own line.Restore the marker line if it was edited. Re-run the bot.
No comments on a PR that should have findingsCheck run-summary.json: did it finish? Did any stage fail? Are there findings in final-findings.json?If POST_MIN_SEVERITY is set to minor or higher, nit findings are dropped. Lower the threshold if needed.
"Waiting author" vote missing or unexpectedCheck VOTE_WAITING_ON env (default none, so no vote by default). Check the max severity in final-findings.json.Adjust the env in your run script or CI pipeline.
Coverage check always says ACs are uncoveredIs the work item linked to the PR? Is the AC text present in the diff or changed files? Is AC_COVERAGE_LLM=1 filtering false positives?Link the work item, or disable AC coverage with AC_COVERAGE_CHECK=0 if it is not applicable for this repo.
Pi timed out or a stage failed with a long diffCheck PI_TIMEOUT_SECS (default 600s) and MAX_DIFF_BYTES (default 200KB). Look at diff.patch size.Split the PR, increase the timeout, or enable chunking.
Container cannot post (403/401 from ADO)The build service identity needs "Contribute to pull requests" on the repo. The ADO token must be current.Grant the permission in Project Settings → Repos → Security → <Project> Build Service.
Findings are too noisy or too strictMount a custom prompt or standards file. Check REVIEW_PROMPT_PATH and REVIEW_STANDARDS_PATH.Tune the markdown files and pass them via -v /host/path:/cfg/file.md -e NAME=/cfg/file.md.
Batch script discovers zero PRsCheck ADO_ORGANIZATION, ADO_PROJECTS, ADO_TARGET_BRANCHES. Verify the PAT can list active PRs.Set the required env vars in the process before invoking the script.

9. Glossary reference

TermMeaning
piThe Pi coding agent CLI, installed globally inside the container. It is the LLM harness used by the review stages.
prb:<key>Idempotency marker at the end of a bot comment. Prevents duplicates when the bot is re-run.
Dry runA run with DRY_RUN=1 (or -DryRun). It produces findings but does not write to ADO.
StageOne discrete step in the 12-step pipeline. Each stage writes a specific artifact file.
ChunkingSplitting a large diff into per-file pieces so each review prompt fits in the model's context window. Controlled by MAX_DIFF_BYTES.
Severity laddernit < minor < major < blocker. Used for filtering, failing, and voting.
AC coverageAcceptance-criteria coverage stage. Checks whether work-item ACs appear to be reflected in the diff.
Stale annotationA reply the bot adds to its old comment when the diff line it referenced is no longer present. The bot does not delete or resolve threads.
Artifact volumeHost directory or Docker named volume where every run's JSON output and logs are stored.
