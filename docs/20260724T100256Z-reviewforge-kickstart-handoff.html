1. Ziel und Erfolgsdefinition

1. Goal and definition of done

Du kannst einen einzelnen Pull Request reproduzierbar reviewen, ohne versehentlich Kommentare zu posten. Du verstehst danach den Artefaktbaum, den Default-Reasoning-Pfad single_pi und die Grenze zwischen Python-Orchestrierung, Pi-Subprozess und Azure DevOps.

You can review one pull request reproducibly without accidentally posting comments. You understand the artifact tree, the default single_pi reasoning path, and the boundary between Python orchestration, the Pi subprocess, and Azure DevOps.

2. Architektur in einem Bild

2. Architecture at a glance

graph TB U[Developer / CI] --> C[reviewforge.cli.main] C --> O[pipeline.orchestrator] O --> P[Stage pipeline] P --> F[Fetch ADO metadata] P --> G[Prepare git repository and diff] P --> R[Execute reasoning engine] R --> S[single_pi default] R --> M[multi_stage explicit legacy] S --> PI[PiCliRunner / pi subprocess] P --> V[Validate anchors] P --> A[PostToAdoStage] A --> API[Azure DevOps REST API] O --> ART[per-run artifacts]

Der Produktionspfad ist bewusst physisch klein: DEFAULT_PIPELINE enthält Fetch, Repository-Vorbereitung, Engine, Anchor-Validierung und Posting. Das Modell liefert niemals ADO-Network-Side-Effects.

The production path is intentionally physically small: DEFAULT_PIPELINE contains fetch, repository preparation, engine execution, anchor validation, and posting. The model never performs ADO/network side effects.

3.1 Umgebung installieren

3.1 Install the environment

uv venv
. .venv/bin/activate
uv sync --extra dev

Zusätzlich erforderlich: git, rg (ripgrep) und die pi-CLI. uv sync --extra dev verwendet das Repository-Lockfile uv.lock und installiert die Entwicklungsabhängigkeiten reproduzierbar.

Also required: git, rg (ripgrep), and the pi CLI. uv sync --extra dev uses the repository lockfile uv.lock and installs development dependencies reproducibly.

3.2 Configure the run

export ADO_ORG="your-org"
export ADO_PROJECT="your-project"
export ADO_REPO_ID="your-repository-id"
export PR_ID="123"
export ADO_AUTH_TOKEN="..."

# Optional: explizit kein Posting
export DRY_RUN=1

Nutze für die vollständige Variablenliste docs/reference/environment-variables.md. Secrets gehören in die Umgebung oder ein lokales Secret-Management, nicht in Prompts oder Artefakte.

Use docs/reference/environment-variables.md for the complete variable list. Keep secrets in the environment or local secret management, never in prompts or artifacts.

3.3 No-post Review ausführen

3.3 Run a no-post review

uv run reviewforge review --no-post
# alternativ:
uv run python -m reviewforge review --no-post

Der Repository-Guide dokumentiert zusätzlich uv run reviewforge review --dry-run. Verwende --no-post als explizite Review-only-Grenze; --dry-run verhindert Posting ebenfalls, druckt aber die Findings als JSON.

The repository guide also documents uv run reviewforge review --dry-run. Use --no-post as the explicit review-only boundary; --dry-run also prevents posting but prints findings as JSON.

7. Guardrails und bekannte Risiken

7. Guardrails and known risks

P1

Die direkte Pi-Subprozess-Grenze PiCliRunner.run_json benötigt besondere Aufmerksamkeit: Timeout, ungültiges JSON, Repair-Call und Token-Parsing sind kritisch.

The Pi subprocess boundary PiCliRunner.run_json needs special attention: timeout, invalid JSON, repair calls, and token parsing are critical.

  • --no-post für lokale Entwicklung verwenden.
  • Keine ADO-Tokens in Prompt-Dateien, Findings oder Logs schreiben.
  • Never write ADO tokens into prompt files, findings, or logs.
  • Vor Änderungen an Stage-Outputs die Pydantic-Schemas und Projektionen prüfen.
  • Before changing stage outputs, inspect the Pydantic schemas and projections.
  • Artifact-Namen und prb:<dedupe-key>-Marker sind Kompatibilitätsverträge.
  • Artifact names and prb:<dedupe-key> markers are compatibility contracts.
  • Bei parallelen Pi-Workers keine Sessions teilen; die bestehenden Stage-Implementierungen erzeugen isolierte Session-IDs.
  • Do not share Pi sessions across parallel workers; existing stages create isolated session IDs.

8. Handoff next steps

  1. Lokalen No-post Smoke-Test ausführen und Run-ID notieren.
  2. Run the local no-post smoke test and record the run ID.
  3. run-summary.json, review-result.json und final-findings.json vergleichen.
  4. Compare run-summary.json, review-result.json, and final-findings.json.
  5. Bei Fehlern zuerst run.log, danach Stage-Name und Artefaktpfad dokumentieren.
  6. For failures, capture run.log, the stage name, and the artifact path first.
  7. Erst nach Review der Findings und der Anchor-Mappings mit post --input posten.
  8. Only post with post --input after reviewing findings and anchor mappings.
